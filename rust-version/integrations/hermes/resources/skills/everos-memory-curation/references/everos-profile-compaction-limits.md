# EverOS Profile Compaction Limits

Use this reference when the user asks to compress or clean EverOS profile sections such as `Basic Information` or `Personality & Traits`.

## Observed durable behavior

EverOS personal `profile` can behave as a Cloud-side aggregate view that is not controlled by the normal memory delete path.

Observed outcomes in this setup:

- `get_memories(memory_type="profile")` returned one large profile containing `profile_data.explicit_info` and `profile_data.implicit_traits`.
- Deleting the profile by its exact id returned HTTP 204 but did not shorten or remove the profile aggregate.
- Batch delete by `user_id=hermes_default` cleared structured `episodic_memory` but left the profile aggregate unchanged.
- Direct group delete against the profile group id returned HTTP 204 but left the profile aggregate unchanged.
- Probe attempts for profile-specific endpoints such as `/api/v1/profile(s)/delete` and `DELETE /api/v1/profile(s)/<id>` returned 404.
- A later aggressive cleanup backed up profile/episodic/agent surfaces, collected the profile id, `profile_data.processed_episode_ids`, `explicit_info[].sources`, `implicit_traits[].sources`, current episodic ids, and current episodic parent ids, then deleted 280 unique ids with zero reported failures. After writing a compact seed and flushing to `status="extracted"`, structured `episodic_memory` shrank to one seed but `get_memories(memory_type="profile")` still returned the same large aggregate (`memcell_count=236`, 22 explicit items, 12 implicit traits, 273 processed ids). This is the strongest observed evidence that the UI/profile aggregate is not regenerated by public memory deletes.

Treat this as a Cloud API limitation unless later official docs expose a profile patch/delete/reset/regenerate endpoint.

## Safe compaction strategy

When the user asks to reduce noisy profile sections:

1. Back up structured memory first:
   - `episodic_memory`
   - `profile`
   - `agent_case`
   - `agent_skill`
2. Delete noisy structured episodic/session data if explicitly requested.
3. Do not promise that `Basic Information` or `Personality & Traits` in the EverOS UI will shrink after delete, even after deleting source ids referenced by the profile.
4. Re-seed only compact durable facts as a short personal memory/session if needed.
5. Reduce future context injection by provider configuration:
   - exclude `profile` from provider `memory_types`;
   - keep `top_k` and `max_context_items` low;
   - disable broad `auto_capture` / `capture_agent_memory` unless explicitly requested.
6. Verify the effective behavior with provider `prefetch`, not only with Cloud profile display.
7. If the user needs the UI profile itself to become short, the most reliable current workaround is a fresh EverOS `user_id` seeded only with compact durable facts; keep the old user id as historical/archived rather than spending more time deleting aggregate sources.

## Reporting language

Be precise:

- Good: "Structured episodic recall is compact and provider prefetch is short; the Cloud profile aggregate still displays old long sections due to current API limits."
- Good: "Deleting profile source IDs succeeded at the public API level, but the UI `Basic Information` aggregate did not change; the next clean-slate option is a new EverOS user_id."
- Bad: "The profile has been deleted" when `get_memories(memory_type="profile")` still returns the old aggregate.
- Bad: "Deleting all source IDs will force profile regeneration"; this was tested and did not hold.

## What not to save as memory

Do not save the one-off profile id, commit SHA, package hash, or exact backup filenames as durable memory. Save only the general behavior and safe workflow.
