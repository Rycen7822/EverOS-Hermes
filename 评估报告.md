## 0. 执行摘要

OpenClaw 官方 EverOS 插件与 EverOS-Hermes 的定位不同：

- OpenClaw 插件是 `context-engine`：核心目标是在宿主生命周期中自动 recall、自动写入，不提供显式 MCP tool surface。
- EverOS-Hermes 同时承担两层职责：Hermes memory provider 的自动 recall/capture，以及 MCP tools 的显式 save/search/get/delete/migrate/verify。
- EverOS-Hermes 当前强调 Cloud v1 安全契约、迁移验证工具、Python/Rust 双运行时 parity；这些能力不应被 OpenClaw 的 legacy/group 兼容逻辑冲淡。

本次对照后，最值得升级的不是“照搬 OpenClaw 架构”，而是围绕自动记忆质量和上下文治理做增强：

1. P0：Agent trajectory capture 应从当前“两条摘要消息”升级为“可保留 `tool_calls` / `tool_call_id` 的结构化轨迹”。这会直接影响 EverOS agent_case / agent_skill 的抽取质量。
2. P0：自动 capture 与导入链路应生成 deterministic `message_id`，减少 retry、重复 hook、导入重跑造成的 raw message 重复。
3. P0：文档与安装态校验需要收口。源码和安装态 MCP 当前均为 13 tools；provider explicit tools 为 8 个，这是不同 surface。README badge 中的 `MCP-9 tools` 已过时，应修正并加入 parity verifier。
4. P1：自动 recall 的 context formatter 应升级为分层 assembler：profile 优先、agent skills/cases 独立、episodes/recent raw 分区、score/quality/maturity 排序、全局预算控制。
5. P1：recent raw recall 可以作为独立小预算、默认关闭或 session-only 的可选能力，用来覆盖 EverOS 异步抽取前的上一轮内容；不能直接把 `raw_message` 混入主 memory budget。
6. P1：Provider prefetch/capture policy 应跳过 reset/heartbeat/过短 query，记录轻量 session state，并暴露 recall/capture 统计。
7. P2：新增 diagnose/health/status 能力，降低 API key、base_url、Cloud v1 endpoint、installed binary、MCP tool count、last write status 等排查成本。
8. 不建议：不要默认引入 legacy fallback、group/sender/object/multimodal API。EverOS-Hermes 当前 Cloud v1 contract 是稳定性核心。

推荐路线：先做 P0 的 agent trajectory + message_id + 文档/安装态 parity；再做 P1 的 context assembler / recent raw / budget；最后做 P2 的 diagnose / health / telemetry。

## 2. OpenClaw 插件的关键设计

### 2.1 插件形态：context-engine，而非 MCP server

OpenClaw 插件的 package metadata 与 `openclaw.plugin.json` 表明它是 `context-engine`。它不是 MCP server，也不是一个显式工具集合；它的目标是接管宿主的上下文注入与记忆写入生命周期。

关键生命周期：

- `bootstrap()`：初始化配置、API client、health/probe。
- `ingest()`：接收消息流并更新 session state。
- `assemble()`：在模型回答前检索相关 memory 并渲染为上下文。
- `afterTurn()`：每轮结束后写入对话/agent trajectory。
- `compact()`：在接近上下文预算时给宿主压缩提示。
- `dispose()`：清理 session state。

对 EverOS-Hermes 的启发：

- Hermes provider 已经有类似 `prefetch()` / `sync_turn()` / `on_session_end()` 的 hook；因此无需重构成 OpenClaw 插件形态。
- 更合理的边界仍是：provider 负责自动 recall/capture；MCP tools 负责显式 search/save/get/delete/import/verify。

### 2.2 Session state：比当前 provider 更细，但不必全量照搬

OpenClaw `engine.js` 使用 session state 记录：

- 2 小时 TTL；
- turnCount；
- savedUpTo；
- last context / last messages；
- heartbeat skip；
- `temp:` / `internal:` ephemeral session skip；
- compact/reset 后的 savedUpTo 重置。

EverOS-Hermes 当前 provider 更轻：

- `prefetch()` 基本只判断 active、auto_recall、query 非空。
- `sync_turn()` 只接收当前 user/assistant 文本。
- capture 有 trivial user skip，但 recall 对“ok/继续/谢谢/重置”等低价值 query 缺少过滤。
- 没有 turnCount、early-turn topK multiplier、session-level recall/capture stats。

建议：引入轻量 ProviderSessionState，而不是复制 OpenClaw 的完整 state machine。可记录：

- turnCount；
- last query hash；
- last recall hit counts；
- rendered chars / estimated tokens；
- last write status / task_id；
- skip reason。

### 2.3 Context assembler：OpenClaw 更重视 LLM 可读性

OpenClaw `subagent-assembler.js` 与 `prompt.js` 的设计较成熟：

- query-aware 检索；
- early turns topK 加倍，上限约 20；
- 自动加入 `profile`；
- agentMode 时加入 `agent_memory`；
- profile / skills / cases / episodic / pending(recent raw) 分区；
- profile 区分 explicit vs implicit；
- skills 标注 confidence / maturity；
- cases 标注 task / approach / insight / quality；
- 先 profile，再 skills/cases，再 episodic/recent；
- XML-like `<memory>` 包装，并加 boundary，降低模型混淆。

EverOS-Hermes 当前 `format_search_context()` 的优点是简单、兼容多种响应 shape，并且默认剥离 vector/embedding。但它缺少：

- section priority；
- score threshold；
- sorting；
- global context budget；
- explicit/implicit 解释；
- skill maturity / case quality 的渲染；
- raw_message 与 consolidated memory 的不同置信度提示；
- 统一 boundary / 防污染说明。

### 2.4 Raw message：OpenClaw 的策略有价值，但 EverOS-Hermes 不能默认照搬

OpenClaw V1 search 把检索拆成两路：

- main request：`episodic_memory` + `agent_memory`，使用 caller topK；
- raw request：`raw_message`，固定小 topK，例如 8；
- 两路并行；raw 失败可降级；raw 渲染到 recent context。

这样做的原因是 EverOS extraction 异步，刚写入的上一轮内容可能尚未进入 episode/profile/agent memory。raw_message 可补这个空窗，但信号低、噪声大，必须独立预算。

EverOS-Hermes 当前情况：

- 自动 recall 默认 `memory_types` 是 `episodic_memory` + `profile`。
- 手动 search 可查 `raw_message`。
- 实际 Cloud 清理中已观察到：删除 synthetic user_id 后结构化 memories 为空，但 raw_messages 仍可能可搜索；该问题已记录到 `problems.md`。

因此不建议默认把 `raw_message` 加进主 recall。更稳妥的是新增可选能力：

- `include_recent_raw=false` 默认关闭；
- `recent_raw_top_k=3..8`；
- `recent_raw_session_only=true`；
- `recent_raw_max_chars` 独立；
- raw 只渲染在 `<recent_context>`，并标注“未合并、低置信度、仅短期参考”。

### 2.5 Agent trajectory：OpenClaw 保留工具链，EverOS-Hermes 当前只是摘要

OpenClaw `convert-agent.js` 的关键点：

- assistant `toolCall` / `tool_use` 转为 V1 `tool_calls`；
- `toolResult` / `tool` 转为 V1 role `tool`；
- `toolCallId` 转为 `tool_call_id`；
- tool arguments 转 JSON string；
- user/assistant 文本会 strip 已注入 memory context 和 channel metadata；
- thinking/image 暂时丢弃；
- tool result 做长度 cap。

OpenClaw `saveAgentV1()` 将这些消息写入 `/api/v1/memories/agent`，交给后端抽取 agent_case / agent_skill。

EverOS-Hermes 当前：

- explicit provider/MCP save 已支持 `scope="agent"`、`role="tool"`、`tool_call_id`。
- 但 provider 自动 `sync_turn()` 只构造两条文本消息：
  - `Task request: ...`
  - `Agent response summary: ... Outcome: completed_or_partial ...`
- 自动 capture 不保留工具调用、工具结果、失败/重试、验证命令、HTTP 状态、文件路径等结构化轨迹。

这会限制 agent memory 的质量。Agent memory 的价值在于复用“任务怎么做”，而不是只记住最终回复说了什么。

## 3. EverOS-Hermes 当前状态判断

### 3.1 已经成熟的部分

当前 EverOS-Hermes 已经不是最小 demo，已有较完整的 Cloud v1 安全边界：

- Cloud v1 endpoint whitelist 与 out-of-scope blacklist 已写入 contract。
- personal / agent scope 明确。
- search/get 类型映射明确：search 用 `agent_memory`，get 用 `agent_case` / `agent_skill`。
- delete 支持 exact memory_id 或 user/session batch delete，且有确认文本安全阀。
- `role=tool` 要求 `tool_call_id`。
- vector/embedding 默认剥离。
- workflow helper 已支持 dry-run / batch / flush / verification。
- Python 与 Rust MCP surface 均为 13 tools。
- provider explicit tools 为 8 个，覆盖 save/search/get/flush/forget/save_and_verify/import_and_verify/verify_session。

这些能力不建议推倒重做。

### 3.2 主要短板集中在“自动记忆质量”和“上下文治理”

OpenClaw 对照后，EverOS-Hermes 的主要短板不是 Cloud endpoint 覆盖，而是：

- 自动写 agent memory 的输入质量偏低；
- 自动 recall 的 context shaping 偏粗；
- raw recent context 与 structured memory 的预算策略尚未分层；
- provider telemetry / diagnose 不够强；
- 文档与发布验证对 tool count 的表述不够收口。

## 4. 详细升级项

### P0-1. 升级 provider agent trajectory capture

优先级：P0
收益：高
风险：中
建议版本：v0.3.0 主线第一项

当前问题：

- `capture_agent_memory=true` 时，provider 自动写 agent scope，但只写两条摘要消息。
- EverOS agent memory 抽取 agent_case / agent_skill 需要任务 intent、approach、tool interaction、failure/retry、verification 等轨迹信号。
- 摘要式写法会丢失 tool_call_id 链和工具结果，使 EverOS 难以学习“为什么这样做、哪里失败、如何修复”。

OpenClaw 参考：

- `convert-agent.js` 保留 assistant tool_calls 与 tool results。
- `saveAgentV1()` 使用 `/api/v1/memories/agent` 写完整 agent turn。

建议实现：

1. 新增 Python 模块：`src/everos_hermes/trajectory.py`。
2. 支持两种输入：
   - 当前 `sync_turn(user_content, assistant_content)` 的 fallback summary；
   - 若 Hermes hook 能提供 full messages，则从 messages 中提取 tool calls / tool results / assistant steps。
3. 输出 V1 AgentMessageItem 列表：
   - user：`role=user, content, timestamp, sender_id=user_id`；
   - assistant：`role=assistant, content, tool_calls?`；
   - tool：`role=tool, tool_call_id, content`。
4. 对 tool output 做严格 cap 与 redaction：
   - 默认每个 tool result 1000-2000 chars；
   - 总 agent trajectory budget 例如 8000-12000 chars；
   - 剥离 API key、token、Authorization、cookie、连接串；
   - 删除 `<everos-context>` / `<memory-context>` / channel metadata。
5. 新增配置：
   - `agent_capture_style`: `summary | trajectory | hybrid`；
   - `agent_tool_output_chars`；
   - `agent_trajectory_max_chars`；
   - `agent_capture_include_tool_results`；
   - `agent_capture_redact_secrets=true`。
6. Rust parity：`rust-version/src/provider.rs` 同步结构化转换。Rust shim 如果仍通过命令行参数传长 user/assistant content，需要评估改为 stdin 或临时 JSON 文件，避免命令行长度限制。

验收测试：

- Python provider：模拟包含 assistant tool_call + tool result 的 messages，确认写入 `/memories/agent` 的 messages 包含 `tool_calls` 和 `tool_call_id`。
- tool result 缺 `tool_call_id` 时本地丢弃或返回 warning，不发送非法 role=tool。
- redaction 测试：API key/token 不进入 payload/log。
- cap 测试：超长 tool output 被截断且保留摘要标记。
- Rust parity：同样覆盖 tool_calls/tool_call_id。
- fake Cloud smoke：确认 endpoint 与 body shape。

注意事项：

- 不要把完整 shell output/file dump 原样写入 EverOS；应摘要/截断。
- 不要把 `<everos-context>` 回写，避免记忆自我污染。
- 如果 Hermes 当前 MemoryProvider hook 无法拿到 tool messages，可先在 `on_session_end(messages)` / `on_pre_compress(messages)` 做 session-level trajectory capture；`sync_turn()` 保持 fallback。

### P0-2. 为自动 capture 与导入链路增加 deterministic message_id

优先级：P0
收益：高
风险：低-中

当前问题：

- OpenClaw 对 personal/agent V1 write 都生成稳定 `message_id`。
- EverOS-Hermes 当前 schema 不禁止额外字段，但 provider 自动写入未生成 `message_id`。
- 背景线程、flush timeout 后重试、import 重跑、session hook 重入都可能造成重复 raw_messages。

建议实现：

1. 新增 `make_message_id(session_id, turn_id, role, content_hash, tool_call_id, index)`。
2. provider `sync_turn()` personal 与 agent messages 都带 `message_id`。
3. workflow import 可选择：
   - 输入已有 `message_id` 则保留；
   - 输入没有时 dry-run 报告将生成 deterministic id；
   - `dedupe=true` 时基于 `message_id` 去重。
4. schema 校验：`message_id` 可选，若存在必须为非空安全字符串，长度限制如 128。
5. Rust parity 同步。

验收测试：

- 同一 session/turn/content 两次生成同一 id。
- 同一 turn 两条相同 content 因 index 不同不碰撞。
- tool_call_id 改变时 id 改变。
- import dry-run 报告 duplicate by message_id。

### P1-1. 重构 context formatter 为 OpenClaw-style memory assembler

优先级：P1
收益：高
风险：中

当前问题：

- `format_search_context()` 简单按 Episodes/Profile/Raw/Agent Cases/Agent Skills 输出。
- `max_items` 是每个 bucket 的上限，不是全局预算。
- personal + agent 两次 prefetch 叠加后可能超出预期。
- 缺少 explicit/implicit、confidence/maturity/quality、score threshold、排序与 section priority。

建议实现：

1. 新增 `src/everos_hermes/context_assembler.py`：
   - parse EverOS response -> normalized buckets；
   - score filtering；
   - bucket priority：profile -> agent_skills -> agent_cases -> episodes -> recent_raw；
   - global budget：max_items + max_chars/token；
   - render XML-like block。
2. Provider prefetch 中使用新 assembler；MCP markdown response 可继续保留旧 `format_search_context()` 或逐步迁移。
3. 新增配置：
   - `context_render_mode`: `markdown | xml`；
   - `max_context_chars`；
   - `max_section_items`；
   - `score_threshold`。
4. 建议渲染形态：

```text
<everos-memory>
  <profile>
  <!-- explicit = user stated; implicit = inferred, use carefully -->
  - (explicit) ...
  </profile>
  <agent_skills>
  - [name] conf=0.82 maturity=0.73: ...
  </agent_skills>
  <agent_cases>
  - task: ...
    approach: ...
    insight: ...
  </agent_cases>
  <episodes>...</episodes>
  <recent_context>...</recent_context>
</everos-memory>
```

验收测试：

- profile 永远先于 episodes。
- explicit/implicit 被正确标注。
- skills 按 maturity/confidence 排序。
- cases 按 score/quality 排序。
- raw 放在 recent_context，且受独立预算限制。
- 全局 `max_context_chars` 生效。

### P1-2. 增加 recent raw recall 的独立搜索与预算

优先级：P1
收益：中-高
风险：中

当前问题：

- EverOS 抽取异步，刚写入的 turn 不一定立刻出现在 episodes/profile。
- OpenClaw 用 raw_message 小预算并行搜索填补空窗。
- EverOS-Hermes 自动 recall 默认不含 raw_message；手动可查。
- Cloud raw_messages delete 残留限制意味着 raw 默认召回可能带来长期噪声。

建议实现：

1. 配置：
   - `include_recent_raw=false` 默认；
   - `recent_raw_top_k=3` 或 `5`；
   - `recent_raw_max_chars=1500`；
   - `recent_raw_session_only=true` 默认。
2. prefetch 时分两路：
   - main search：episodic/profile/agent_memory；
   - raw search：`memory_types=[raw_message]`，top_k 小，timeout 短；
   - raw search 失败只记录 warning，不让主 recall 失败。
3. raw result 渲染为 `recent_context`，并提示“unconsolidated, lower confidence”。

验收测试：

- 默认不发 raw search。
- 开启后发第二个 raw search，top_k 独立。
- raw search failure 不影响 main context。
- session_only 时 filters/session_id 生效。

### P1-3. Recall/capture policy：跳过低价值 query 与 reset prompt

优先级：P1
收益：中
风险：低

当前问题：

- `prefetch()` 只检查 query 非空。
- OpenClaw 会跳过 session reset prompt、heartbeat、短 query。
- 对 Hermes 来说，很多“ok/继续/谢谢/是的”不需要 recall。

建议实现：

1. 复用/扩展 `_TRIVIAL_RE` 到 prefetch。
2. 新增：
   - `min_recall_query_chars`，默认 3 或 5；
   - `skip_recall_patterns`，覆盖 `/new` reset prompt、heartbeat/system-like prompt；
   - `recall_cooldown_ms`，避免同一 query 重复检索。
3. 可选 early-turn topK multiplier：`early_turn_top_k_multiplier=2`，上限 20；默认关闭或只在实验配置下开启。

验收测试：

- `prefetch("ok")` 不调用 search。
- reset prompt 不调用 search。
- early turn topK 按配置提高但不超过上限。

### P1-4. Context budget 与 token 成本控制

优先级：P1
收益：中
风险：低

当前问题：

- `max_context_items` 是 bucket-level，不能精确约束 token。
- personal + agent recall 叠加可能增加上下文。
- 用户当前偏好是：自动写 Agent Memories 开启，但 agent recall 默认关闭，以控制上下文噪声。

建议实现：

1. 新增配置：
   - `max_context_chars`；
   - `max_context_tokens_estimate`；
   - `max_section_items` map，如 profile=5, skills=3, cases=3, episodes=5, recent=3。
2. assembler 记录并可在 debug/diagnose 中显示：
   - hit_counts；
   - rendered_chars；
   - estimated_tokens；
   - truncated_sections。
3. 若 `agent_recall=true`，agent section 应有更小默认预算，避免 agent cases/skills 抢占 profile 与用户偏好。

验收测试：

- 超预算时低优先级 section 被裁剪。
- profile explicit_info 优先保留。
- agent_recall 开启后总预算不翻倍。

### P2-1. Diagnose / health / telemetry 工具

优先级：P2
收益：中
风险：低

OpenClaw bootstrap 做 health check 与 apiVersion probe。EverOS-Hermes 不应默认 fallback legacy，但可以把 probe 变成诊断工具。

建议新增：

- MCP tool：`everos_diagnose`；或 provider tool：`everos_memory_diagnose`。
- 检查项：
  - API key 是否存在，不打印 key；
  - base_url 是否可达；
  - `/api/v1/memories/get` 最小 probe 是否 routed；
  - 当前 configured user_id；
  - provider config 摘要，必须 redacted；
  - installed binary version；
  - MCP tool count；
  - provider explicit tool count；
  - last provider write/recall status。

验收测试：

- 无 key 时只返回配置错误，不泄露环境。
- fake server 下可报告 endpoint routed。
- 输出中不包含 Authorization/API key。

### P2-2. Background failure 可观测性增强

优先级：P2
收益：中
风险：低

当前 Python provider 已有 redacted error log；Rust shim `_spawn()` 把 stdout/stderr 丢弃。建议：

- Rust shim background 子进程 stderr 写入 `$HERMES_HOME/everos.log`，保持 redacted。
- provider success path 记录轻量 metrics：operation/scope/task_id/status/latency/hit_count/rendered_chars。
- 在 diagnose 中展示 last_status。

### P2-3. API version probe 仅作为诊断，不作为默认 fallback

OpenClaw 的 `apiVersion=auto` 是面向自托管/legacy 后端的兼容层。EverOS-Hermes 当前 contract 明确 Cloud v1，并显式排除 v0/legacy。

建议：

- 不默认实现 legacy fallback。
- 若用户配置 `compatibility_probe=true`，仅诊断并给出建议。
- 不自动切换到 legacy endpoint，以免破坏 Cloud v1 contract 与 tests。

### P2-4. 不建议升级项：不要默认引入 group/sender/object/multimodal

OpenClaw 代码中有 `groupId`、soloGroupId、legacy group 相关兼容逻辑。但 EverOS-Hermes 当前 contract 已明确排除：

- group memory；
- groups/senders CRUD；
- multimodal object signing/storage。

建议继续保持排除，除非未来单独开 v0.3+/v0.4 milestone，并给出完整 API contract、tests、release note。不要因为 OpenClaw 有相关字段就混入默认 MCP/provider schema。

## 5. 推荐实施路线

### 阶段 A：P0 质量补强

目标：不改变默认 recall 行为，不扩大 Cloud endpoint；先提高自动 agent memory 质量、写入幂等性、发布验证清晰度。

任务：

1. `trajectory.py`：结构化 agent trajectory extractor。
2. provider `sync_turn` / `on_session_end` 支持 trajectory/hybrid capture。
3. deterministic `message_id`。
4. Python tests + Rust parity tests。
5. README badge/tool count 修复。
6. installed/source parity verifier：分别验证 MCP 13 与 provider 8。

验收：

- Python 全量测试通过。
- Rust fmt/clippy/tests 通过。
- fake Cloud 验证 `/memories/agent` body 包含 `tool_calls` / `tool_call_id`。
- secret scan/redaction 通过。
- installed MCP verifier 显示 13 tools。
- installed provider verifier 显示 8 tools。

### 阶段 B：P1 recall 质量与预算

目标：提高自动 recall 的可用性，同时控制 token 噪声。

任务：

1. context assembler v2。
2. section budgets / global char budget。
3. optional recent raw recall。
4. prefetch skip policy。
5. docs：解释 `agent_recall` / `include_recent_raw` 的 token 成本与默认关闭原因。

验收：

- 单元测试覆盖排序、预算、raw 失败降级。
- 在当前用户配置下默认仍不自动召回 agent memory，除非显式开启。
- 不把 raw_message 默认加入主 memory_types。

### 阶段 C：P2 运维诊断

目标：降低安装/配置/Cloud 状态排查成本。

任务：

1. diagnose tool/subcommand。
2. Rust shim background log。
3. installed-vs-source parity 纳入 release checklist。
4. 只读 health/probe 文档化。

## 6. 与当前用户环境的建议

当前用户环境已开启 automatic Agent Memories writing，但 agent recall 仍关闭。结合本次评估，建议保持：

- `capture_agent_memory=true`：可以继续写。
- `agent_recall=false`：暂时保持，等 context assembler/budget 完成后再考虑开启。
- `include_recent_raw=false`：在 raw delete 残留问题没有 Cloud 侧清晰解法前，不建议默认开启。
- 需要 agent/raw memory 时，优先显式调用 MCP/provider search。

如果马上做一个小版本，最值得优先做的是：

1. 自动 agent trajectory 从摘要式升级为结构化。
2. deterministic `message_id`。
3. README badge/tool count + installed/source parity verifier。

## 7. 风险与边界

1. Tool trajectory 可能含秘密、长日志、文件片段。必须先实现 redaction/cap，再扩大自动捕获。
2. raw_message recall 有帮助，但也最容易噪声化；必须独立小预算且默认关闭或 session-only。
3. context assembler 不能让 prompt 过重；应保留全局 budget。
4. Rust shim 通过命令行传长 user/assistant content，未来 trajectory JSON 可能超过命令行限制；需要考虑 stdin/temp file 协议。
5. 不要复制 OpenClaw legacy/group 逻辑到默认路径；EverOS-Hermes 的 Cloud v1 contract 是当前稳定性的核心。
6. installed provider tools 与 MCP tools 是不同 surface：provider 8 不代表 MCP 缺 5 个工具。

## 8. 优先级表

| 优先级 | 升级项 | 收益 | 风险 | 推荐状态 |
|---|---|---:|---:|---|
| P0 | 结构化 agent trajectory capture | 高 | 中 | 立即规划 |
| P0 | deterministic message_id / idempotency | 高 | 低 | 立即规划 |
| P0 | README/tool count/installed parity verifier | 中 | 低 | 立即修复 |
| P1 | OpenClaw-style context assembler | 高 | 中 | P0 后做 |
| P1 | optional recent raw recall 独立预算 | 中高 | 中 | 默认关闭实验 |
| P1 | recall/capture skip policy | 中 | 低 | 与 assembler 同步 |
| P1 | global context token/char budget | 中 | 低 | 与 assembler 同步 |
| P2 | diagnose/health/status tool | 中 | 低 | 后续 DX |
| P2 | Rust shim background log | 中 | 低 | 后续 DX |
| 不建议 | 默认 legacy fallback / group/sender/object 支持 | 不确定 | 高 | 保持排除 |

## 9. 结论

OpenClaw 插件最值得 EverOS-Hermes 借鉴的不是 installer，也不是 legacy/group 兼容，而是三点：

1. 自动记忆应写入高质量的 agent trajectory，而不是只写最终回答摘要。
2. 自动 recall 应按 memory 类型分层组织，并用预算控制上下文成本。
3. recent raw context 可以补 EverOS 异步抽取空窗，但必须独立小预算、可关闭，并明确低置信度。

EverOS-Hermes 当前 Cloud v1 contract、MCP workflow helper、Python/Rust parity 已经比较扎实。下一轮升级应避免扩大 endpoint 面，集中补“自动记忆质量”和“上下文治理”。在当前环境下，建议继续保持 agent memory 自动写入开启、agent recall 关闭，等 trajectory capture 与 context budget 完成后再评估是否打开 agent recall。
