## 0. 规划结论

本轮升级的目标是把 EverOS-Hermes 从“agent scope 已能写入/flush”推进到“Agent Memories 生命周期可审计、可验证、可发布”。升级完成后，所有 Python MCP、Python provider、Rust MCP、Rust provider 都必须显式区分以下状态：

1. `agent_raw_queued`：agent-scope raw messages 已被 Cloud 接收或排队。
2. `agent_flush`：agent flush 已执行、超时、瞬时失败、跳过，且 retry 语义明确。
3. `agent_structured_visible`：`memory_types=["agent_memory"]` 搜索、`agent_case` get、`agent_skill` get 的结构化可见性结果。
4. `agent_visibility_status`：`visible`、`not_visible`、`partial`、`unchecked`、`error` 中的一个稳定值。
5. `agent_visibility_checks`：每次验证的 query、memory_type、endpoint 类别、hit_count、status、latency_ms、error。每个 check 必须有 `latency_ms`，失败 check 也必须记录耗时。

升级不得把 queued、flush success、provider captured、workflow verified 混同为 Agent Memories structured 可见。后续任何执行、压测、发布报告都必须单独报告 raw queued、flush status、structured visible 三层。

## 2. 目标契约

### 2.1 统一可见性报告结构

新增内部结构 `AgentVisibilityReport`，Python 使用 `dict[str, Any]` 生成稳定 JSON；Rust 使用 `serde_json::Value` 生成同构 JSON。所有 workflow/provider/MCP 返回中的字段名、类型和值域必须一致。

字段固定为：

```json
{
  "agent_raw_queued": true,
  "agent_flush": {"ok": true, "status": "success", "retryable": false},
  "agent_structured_visible": false,
  "agent_visibility_status": "not_visible",
  "agent_visibility_checks": [
    {"kind": "search", "memory_types": ["agent_memory"], "query": "...", "hit_count": 0, "status": "miss", "latency_ms": 0.0},
    {"kind": "get", "memory_type": "agent_case", "hit_count": 0, "status": "miss", "latency_ms": 0.0},
    {"kind": "get", "memory_type": "agent_skill", "hit_count": 0, "status": "miss", "latency_ms": 0.0}
  ]
}
```

值域固定为：

- `agent_raw_queued=true` 只表示 add/import/save 接收成功或 batch 部分成功。
- `agent_raw_queued=false` 表示 agent-scope write 没有排队成功。
- `agent_raw_queued=null` 表示本次 workflow/provider action 没有执行 agent write。
- `agent_structured_visible=true` 表示至少一个 agent structured check hit_count > 0。
- `agent_structured_visible=false` 表示已执行 agent structured checks 且全部 hit_count 为 0。
- `agent_structured_visible=null` 表示没有执行 structured checks。
- `agent_visibility_status="visible"`：所有必选 structured checks 都有命中。
- `agent_visibility_status="partial"`：至少一个 structured check 命中，同时至少一个 structured check miss 或 error。
- `agent_visibility_status="not_visible"`：所有已执行 structured checks 都为 0 命中。
- `agent_visibility_status="unchecked"`：没有执行 structured checks。
- `agent_visibility_status="error"`：所有 structured checks 都失败，或本地参数错误导致 audit 无法启动。单个 search/get error 不得中断其他 checks。

### 2.2 新增配置

Python `EverOSMemoryProvider.DEFAULT_CONFIG` 与 Rust `ProviderConfig::default()` 同步新增：

```json
{
  "agent_visibility_verify_after_write": false,
  "agent_visibility_verify_after_flush": false,
  "agent_visibility_queries": [],
  "agent_visibility_top_k": 5,
  "agent_visibility_timeout": 30,
  "agent_visibility_get_page_size": 20,
  "agent_visibility_retry_flush_attempts": 1,
  "agent_visibility_retry_flush_backoff_ms": 250
}
```

含义固定为：

- 默认不开启 automatic post-write visibility checks，避免每 turn 增加 Cloud 负载。
- Workflow helpers 的显式 verify 接口不受默认关闭影响；调用 `save_and_verify`、`import_and_verify`、`verify_session_ingest` 且 `scope="agent"` 时必须返回 visibility report。
- `agent_visibility_verify_after_flush=true` 时，provider agent flush 后同步执行一次 visibility audit。
- `agent_visibility_queries` 为空时，使用 agent write content 的前 200 字符、trajectory source、session id marker 组合生成 query；不得生成空 query。
- flush retry 只处理 transient network send failure 与 timeout；Cloud 4xx schema/validation error 不 retry。

### 2.3 Agent visibility helper API

Python 新增文件：`src/everos_hermes/agent_visibility.py`。

公开函数固定为：

```python
def audit_agent_visibility(
    *,
    client: EverOSClient,
    user_id: str,
    session_id: str | None,
    queries: list[str],
    top_k: int = 5,
    timeout: float | None = None,
    get_page_size: int = 20,
) -> dict[str, Any]: ...

def build_agent_visibility_report(
    *,
    agent_raw_queued: bool | None,
    agent_flush: dict[str, Any] | None,
    checks: list[dict[str, Any]],
) -> dict[str, Any]: ...
```

Rust 新增文件：`rust-version/src/agent_visibility.rs`。

公开函数固定为：

```rust
pub fn audit_agent_visibility(
    client: &EverOSClient,
    user_id: &str,
    session_id: Option<&str>,
    queries: Vec<String>,
    top_k: i64,
    timeout: Option<f64>,
    get_page_size: u64,
) -> Result<Value, EverOSError>;

pub fn build_agent_visibility_report(
    agent_raw_queued: Option<bool>,
    agent_flush: Option<Value>,
    checks: Vec<Value>,
) -> Value;
```

Rust helper 返回语义固定为：check-level HTTP/search/get error 写入 `agent_visibility_checks` 并返回 `Ok(report)`；只有本地参数无效或调用前状态不满足时返回 `Err(EverOSError)`。

Audit 顺序固定为：

1. 对每个非空 query 执行 `search_memories(memory_types=["agent_memory"], method="hybrid", include_original_data=false, include_vectors=false)`。
2. 执行 `get_memories(memory_type="agent_case", page=1, page_size=agent_visibility_get_page_size)`。
3. 执行 `get_memories(memory_type="agent_skill", page=1, page_size=agent_visibility_get_page_size)`。
4. 每个 check 独立记录 `status`、`hit_count`、`latency_ms`、`error`；单个 check 失败不得吞掉其他 check。
5. 所有 check 完成后统一计算 `agent_structured_visible` 与 `agent_visibility_status`。

### 2.4 Flush retry 契约

Python `EverOSClient` 与 Rust `EverOSClient` 不在底层 `request_json()` 默认重试所有请求。重试放在 workflow/provider 的 flush helper 中，避免对非幂等 add/import 写入造成重复。

新增内部 helper：

- Python：`flush_memories_with_retry()` 放在 `src/everos_hermes/workflows.py` 或新建 `src/everos_hermes/retry.py`。若多个模块共用则新建 `retry.py`。
- Rust：`flush_memories_with_retry()` 放在 `rust-version/src/workflows.rs` 或新建 `rust-version/src/retry.rs`。若 provider 和 MCP 都调用则新建 `retry.rs`。

重试规则固定为：

- 只重试 `EverOSTimeoutError` / `EverOSError::Timeout` 和 request-send/network 类错误。
- 最多 `1 + agent_visibility_retry_flush_attempts` 次实际调用。
- 每次失败都进入返回 payload 的 `attempts` 列表。
- Cloud 400/401/403/422 不 retry，直接返回 error payload。
- 最终失败 payload 必须含 `retryable=true`、`operation="flush"`、`scope="agent"|"personal"`、`attempt_count`。

### 4.5 Real Cloud smoke

真实 Cloud smoke 只在所有单元、parity、fake-server smoke 通过后执行。

硬约束：

- 使用已有非默认 user id：`hermes_mcp_stress_main_20260512_125809`。
- session id 使用新 synthetic marker：`agent_visibility_upgrade2_<YYYYMMDD_HHMMSS>`。
- 不新建 user id。
- 不触碰 `hermes_default`。
- 每次写入都包含 marker，cleanup 只删除该 synthetic session。

最小真实 Cloud smoke 步骤：

1. `everos_save_and_verify(scope="agent", content=<marker>, verification_query=<marker>)`。
2. `everos_import_and_verify(scope="agent", batch_size=5, messages=[assistant/tool 带 tool_call_id], verification_queries=[marker])`。
3. `everos_flush_memories(scope="agent", session_id=<synthetic>, timeout=45)`。
4. `everos_search_memories(memory_types=["agent_memory"], query=<marker>, session_id=<synthetic>, top_k=10)`。
5. `everos_get_memories(memory_type="agent_case", session_id=<synthetic>, page_size=20)`。
6. `everos_get_memories(memory_type="agent_skill", session_id=<synthetic>, page_size=20)`。
7. `everos_delete_memories(user_id=<existing_non_default>, session_id=<synthetic>, confirm=true, confirm_scope_text="delete user_id=<id> session_id=<synthetic>")`。
8. 等待 30 秒后复核 structured search/get；若 structured late arrival 出现，对同一 user/session 再执行一次 delete。
9. raw residual 单独记录；raw residual 不阻断 structured cleanup gate。

真实 Cloud smoke 通过标准：

- 工具返回中三层状态均存在：raw queued、flush、structured visible。
- 若 Cloud 仍返回 agent structured 0，执行了 visibility audit 的 workflow 状态必须是 `agent_not_visible`，不得是 `verified` 或 `not_yet_searchable`。
- 若 Cloud 返回 agent structured 命中，workflow 状态必须是 `verified` 或 `partial`，且 hit_count 记录准确。
- cleanup 后 structured prefix search/get 为 0。
- raw residual 单独写入 `problems.md` 或 `升级note.md`，不写成 cleanup 失败。

### 4.6 长程压力回归

本轮功能通过最小 real Cloud smoke 后，执行 30-60 分钟 targeted pressure。只有在用户要求或发布前 gate 需要时执行 3 小时压力。

targeted pressure 必须覆盖：

- agent save/add/import/save_and_verify/verify_session。
- provider sync_turn、on_pre_compress、on_delegation、on_session_end。
- agent visibility audit 全 miss、partial hit、visible hit 的 fake-server 分支。
- real Cloud 分支记录真实 hit/miss，不把 0 命中当本地失败。
- flush transient retry 计数。
- cleanup structured 与 raw residual 分离。
